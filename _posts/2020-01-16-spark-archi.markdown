## 스파크의 등장  
<center><img src="../assets/img/spark/spark.png" style="width:300px;"></center>

스파크는 UC버클리의 마태 자하리아가 개발한 **``오픈소스 분산 쿼리 및 처리 엔진``**입니다. 2012년에 처음 배포되었으며, 코드는 아파치 재단에 기부되었습니다. 스파크는 맵리듀스를 대체할 수 있는 리소스 스케줄링으로 설계 되어 기존의 MapReduce에 비해 유연성과 확장성이 좋아서, 속도가 빠릅니다. 

**스파크는 무엇을 할 수 있을까요?**
- 데이터를 읽고, 변형하고, 합하고, 통계 모델을 학습할 수 있습니다. 

**스파크의 프로그래밍 인터페이스**
- 스파크는 본래 scala로 작성되어 jvm에서 실행되나
- 다양한 프로그래밍 인터페이스를 지원한다.
- scala, python, java, sql , r 등

**스파크가 제공해주는 라이브러리에는 무엇이 있을까요?**
 - 자체 데이터프레임을 제공합니다.  
 - SQL에서 사용하는 쿼리   
 - MLib(머신러닝)   
 - GraphX(그래프처리)  
 - 스파크 스트리밍
 
## 스파크 RDD  

- 스파크 RDD: 스파크 응용 프로그램의 기본 데이터 추상화 구조이다.
- 클러스터에 분산된 인 메모리 데이터 모음이라고 할 수 있음. 
- 스파크는 맵리두스에 비해 RDD를 사용함. RDD는 다음 특성이 있음
    - 분산형
    - 내결함성 fault-tolerant
    - 인메모리 in-memory: 인메모리 구조를 재사용하여 메모리 사용을 극대화함. 

## 스파크의 구조 : 클러스터 환경

### 로컬 모드/클러스터 모드

스파크는 한대의 서버에서 동작하게 할 수도 있고, 여러대의 서버에서 동착하게 할 수도 있습니다.
- 로컬 모드: 한대의 서버에서 동작  
- 클러스터 모드: 여러대의 서버에서 동작  

클러스터란 여러대의 서버가 네트워크를 통해 하나의 서버인 것처럼 동작하는 것입니다. 클러스터 모드를 사용하기 위해서는, 여러 서버를 제어하는 모듈인 클러스터 매니저가 필요합니다.스파크의 경우 자체 구현한 클러스터 매니저를 제공하며, 외부 클러스터 매니저를 사용할수도 있습니다. 
- 클러스터 매니저: 여러 서버를 제어하는 모듈  

드라이버와 executor  
- 드라이버: 메인 함수를 실행, sparkcontext를 생성하는 프로그램  
- Executor: 실제 데이터 처리를 담당하는 프로세스

### 마스터 서버/슬레이브 서버

클러스터 모델은 클러스터를 구성하는 전체 서버를 관리하는 서버와 실제 작업을 처리하는 서버로 나누어 운용합니다. 

- 마스터 서버 = 매니저 : 클러스터 운영, 제어에 사용되는 서버
- 워커(worker) = 슬레이브 : 실제 작업에 사용되는 서버

마스터/슬레이브의 구분은 물리적인 구분을 의미하는 것이 아닙니다. 마스터/슬레이브 서버는 물리적으로 하나의 서버에서 구동될 수도 있습니다. 스파크의 클러스터 모드에서 '클러스터 매니저'는 마스터 서버에 속한다고 할 수 있습니다. 스파크의 클러스터 모드에서 여러개의 '워커 노드' 들은 슬레이브 서버에 속한다고 할 수 있습니다.

### 스파크 응용프로그램의 구조  

<center><img src="../assets/img/spark/spark-archi.png" style="width:70%;"></center>
<p></p>

|구성 요소|역할|포트|
|-----|---|---|
|클러스터 매니저| **리소스를 담당**. 리소스와 동작을 제어 - 워커 노드를 모니터링하고 예약함 |8080| 
|스파크 드라이버| **응용프로그램을 스케줄링**. 메인 함수로 sparkcontext를 생성하여 워커 노드에게 task를 지시함 |4040|
|워커 노드| **Executor 프로세스**를 호스팅하는 노드||
|Executor|스파크 작업의 개별 task, 즉 dag 작업을 실행함||

**1. 클러스터 매니저**  

클러스터 매니저는 스파크 클러스터 내에서 `마스터`역할을 수행하는 컴포넌트입니다. 클러스터 매니저는 **전체 서버의 리소스와 동작을 제어하는 모듈**입니다. 마스터 호스트:8080에 접속하면 스파크 마스터 프로세스의 웹 UI를 제공받을 수 있습니다.

- 클러스터 매니저의 기능
  - **전체 리소스를 제어함.**
  - **worker 노드를 모니터링**
  - **worker 노드의 리소스를 예약**
- 스파크는 여러 종류의 클러스터 매니저를 지원합니다. 
  - 스파크 자체 제공되는 스탠드얼론
  - 하둡의 YARN
  - Mesos   


**2. 스파크 드라이버**  

스파크에서 드라이버 프로그램은 **메인 함수를 가지고 있으면서 SparkContext를 생성하고 관리하는 프로그램**입니다. 스파크 애플리케이션을 실행하면 이 스파크 드라이버 프로그램 속 메인 함수가 실행되고, 메인 함수가 실행되면 SparkContext가 실행됩니다. SparkContext는 이어서 워커 노드의 executor 프로세스를 구동시켜 잡을 수행시키게 됩니다. 이를 통해 스파크 드라이버는 **워커 노드에게 작업을 지시하고 결과를 취합하는 역할**을 수행합니다.  

- 드라이버의 기능
  - **응용 프로그램을 task로 변환**   
  스파크 드라이버는 응용 프로그램을 task로 변환해 줍니다. task란 하나의 노드에서 실행되는 물리적 실행 단위입니다. 

  - **응용 프로그램 DAG 작성 및 계획**   
  드라이버는 응용 프로그램을 어떻게 실행할지 계획하기 위해 DAG(directed acyclic graph)를 작성하고, 작성된 DAG에 대해서 물리적인 실행 계획으로 변환합니다. 

  - **응용 프로그램 스케쥴링**   
  드라이버는 executor 프로세스에서의 개별 작업들을 위한 스케쥴링을 합니다. executor들은 task를 실행하고 rdd 데이터를 저장하는 프로세스인데요, 드라이버는 실행중인 executor를 살피고 각 task가 적절한 위치에서 실행될 수 있도록 노력합니다. 

  - **응용 프로그램의 결과 반환**  
  드라이버는 클라이언트 측에 결과 데이터를 반환합니다. 포트 4040에 응용 프로그램 UI를 제공합니다. 4040 포트로 진입하면 각 응용 프로그램 별 stage, task 등을 확인할 수 있습니다. 

**드라이버**가 응용 프로그램 단위에서 조정을 하는 역할을 한다면, **클러스터 매니저 (마스터)**는 클러스터 전체 내에서 리소스를 관리하는 역할을 합니다. 

**3. SparkContext**  

스파크 공식 문서에 따르면, SparkContext는 **스파크의 시작점 또는 스파크 클러스터와의 연결**을 나타냅니다. 즉, SparkContext를 준비시키는 것은 클러스터 매니저와의 연동 등 스파크 애플리케이션의 동작에 필요한 서비스를 준비시키는 것입니다. 하나의 스파크 애플리케이션에서는 오직 하나의 SparkContext만을 생성시킬 수 있습니다. 

**4. 작업자 Worker**

executor 프로세스를 호스팅하는 노드를 작업자 worker 노드라고 합니다. 여기에는 유한하거나 고정된 수의 실행자가 특정 시점에 할당 됩니다. 

**5. 실행자 Executor** 

스파크 프로그램을 실행하는 것은 곧 스파크 job(=task=dag작업)을 실행하는 것입니다. executor 프로세스는 **스파크 작업의 개별 task, 즉 dag 작업을 실행하는 작업 실행 프로세스**입니다. 실행자는 작업자의 cpu, 메모리 등 리소스를 예약하기도 합니다. 

- Executor의 기능

  - 애플리케이션을 구성하는 작업들을 실행하여 결과를 드라이버에 반환합니다.
  - 블록 매니저를 통해 rdd 저장 메모리를 제공합니다. 

- 응용 프로그램에 특화된 프로세스이며, 그 응용 프로그램이 완료되면 종료됩니다. 
- 스파크 프로그램 하나는 여러개의 많은 Executor로 구성되어 있으며 병렬로 작업합니다. 
- Executor의 JVM에는 전용 메모리 공간인 heap이 할당됩니다. 
- 드라이버는 응용프로그램 전체를 관장하며, 워커 노드 및 Executor는 할당된 작업만 관장합니다. 

## 스파크 응용프로그램의 수행 절차

<center><img src="../assets/img/spark/spark-archi.png" style="width:70%;"></center>
<p></p>

1. 스파크 애플리케이션의 드라이버 프로그램이 실행됩니다.  
2. SparkContext가 생성되면서 클러스터 매니저와 연동됩니다. 
3. 각 클러스터 서버에 작업을 처리하기 위한 프로세스가 생성됩니다. = 즉, 작업을 수행하는 서버인 워커 노드에 executor 프로세스가 생성됩니다.
4. 드라이버 프로그램은 task(job)을 스케줄링합니다. 
5. task는 여러 executor에 할당됩니다. 
6. executor는 1) task를 처리하고 2) 처리된 데이터를 재사용을 위해 메모리에 저장해 둡니다. 


## 스파크와 연동된 YARN(리소스 매니저=클러스터 매니저)

YARN은 하둡의 데이터 처리를 제어하고 조율하는 ``프로세스 스케줄러``입니다. YARN에서 실행되는 스파크 응용 프로그램의 배포모드는 클라이언트 모드와 클러스터 모드가 있습니다. 
 
### 클라이언트 디플로이 모드  

응용 프로그램의 드라이버 프로세스가 클라이언트에서 실행된다.  

1. 클라이언트는 YARN(리소스 매니저)에 응용프로그램을 제출한다. 드라이버, 스파크세션이 만들어져서 클라이언트에서 실행된다.  
2. YARN(리소스 매니저)은 응용 프로그램에 대해 스파크 마스터를 할당한다.  
3. 스파크 마스터는 YARN에게 실행자로 사용될 컨테이너를 요청한다. 그러면 실행자가 생성된다.  
4. 클라이언트에 속한 드라이버는 실행자와 통신한다.  
5. 드라이버는 클라이언트에 진행률과 결과를 보고한다.    

### 클러스터 디플로이 모드

응용 프로그램과 드라이브는 마스터의 하위 프로세스가 되어 클러스터에서 실행된다. 만약 마스터가 실패하면 다른 노드에서 다시 인스턴스화될 수 있다. 

1. 클라이언트는 YARN(클러스터 매니저)에 응용프로그램을 제출한다.  
2. YARN(클러스터 매니저)는 응용 프로그램에 대해 스파크 마스터를 할당한다.  
3. 마스터와 동일한 노드에서 드라이버 프로세스가 생성된다.  
4. 마스터는 YARN(클러스터 매니저)에게 실행자용 컨테이너를 요청한다.  
5. 실행자는 마스터에 할당된 컨테이너 안에 생성된다.  
6. 드라이버는 실행자와 통신한다.  
7. 드라이버는 클라이언트에 진행률과 결과를 보고한다.
